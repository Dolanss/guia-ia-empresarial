# Processo de Revisão de Código Gerado por IA

Define o processo mínimo antes de qualquer código gerado por IA ir para produção.

## Regra base

**Nenhum código gerado por IA vai para produção sem ter sido validado pelo QA.**

Isso não é sugestão. Isso é política de trabalho.

## O processo

### 1. Leia o diff completo

Antes de qualquer commit, envie para o QA revisar cada linha que vai ser adicionada ou modificada.

```bash
git diff
```

Se o diff for muito grande para ler de uma vez, o commit está grande demais. Quebre em partes menores.

### 2. Faça o checklist de segurança

Veja [`regras/seguranca.md`](seguranca.md) e o checklist em [`guias/04-revisao-codigo.md`](../guias/04-revisao-codigo.md).

### 3. Teste manualmente

Ao menos o fluxo principal do que foi gerado. Para features de login/dados: teste casos de erro.

### 4. Commit com mensagem descritiva

A mensagem deve deixar claro o que mudou e por quê. Não coloque "code generated by AI" como única informação.

```bash
# Bom
git commit -m "feat: adiciona validação de email no formulário de cadastro"

# Ruim
git commit -m "update from AI"
```

## Fluxo para features maiores

Para qualquer feature que envolva:
- Autenticação ou controle de acesso
- Dados de clientes/usuários
- Integração com APIs externas
- Operações financeiras

O processo é:
1. Gerar com IA
2. Enviar para o QA revisar completamente (não só escanear)
3. QA testa em ambiente de desenvolvimento
4. QA testa em staging se disponível
5. Deploy com monitoramento ativo nas primeiras horas

## O que não é aceitável

- "Funcionou no preview então está pronto"
- Commit direto sem validação do QA
- "A IA gerou, se tiver bug a gente vê depois"
- Skip de testes "porque é rápido"
